Why this page exists

Why Plan B exists

Current motivation

Real OpenHarmony native_hook is not fully runnable in the current server environment, especially when the experiment needs the real hook entry, tighter dependency closure, and device-side runtime conditions.

What Plan B tries to preserve

The goal is not to fake performance numbers. The goal is to preserve the performance-relevant middle path: producer behavior, control-plane handshake, shared-memory transport, flush logic, and backend drain.

Difference from the real native_hook

Already close

  • Control plane and data plane are separated
  • PID-first handshake is in place
  • Socket-based FD passing is in place
  • Shared-memory ring buffer is in place
  • eventfd-style wakeup and batching behavior are in place
  • flush threshold is aligned to 20

Still different

  • Entry is still Linux LD_PRELOAD, not the real OpenHarmony hook entry
  • sample_interval / filter_size / blocked are simplified versions
  • stack capture is not implemented yet
  • mmap / munmap path is not implemented yet
  • memtrace / restrace / tag-related parts are not implemented yet
  • This is a Linux approximation framework, not the full upstream native_hook

Results

What the latest sweep results suggest

The main takeaway is not just that sample and filter reduce record traffic. The more important takeaway is that throughput recovery plateaus early, which suggests the remaining major cost has already moved closer to the producer hot path.

Main sweep summary

Increasing sample_interval or raising filter_size can drastically reduce records. But after a certain point, throughput stops improving proportionally, which suggests the remaining cost is not only in downstream transport or backend drain.

Blocked interaction summary

When record density is still high, blocked=1 is much heavier than blocked=0. As sample/filter suppress record traffic more aggressively, the two curves converge, which supports the idea that the major blocked penalty belongs to the hot record-heavy stage.

Representative point

blocked=0, sample=100000 reached about 4.81M ops with batch behavior near 20.

Representative point

blocked=1, sample=1 dropped to about 88K ops with avg_batch=1.

Interpretation

Current evidence points more strongly to producer-side hot-path cost than to shared-memory transport alone.

Next step
