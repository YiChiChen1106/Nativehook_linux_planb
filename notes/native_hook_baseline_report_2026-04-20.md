## 4. 结果指标说明

结果文件中的关键字段为：

```text
The total g_times(malloc/free): XXXXX
```

其含义是：

- 在设定的测试时长内，程序总共完成的 `malloc/free` 次数

由于本次所有实验时长均为 `60` 秒，因此吞吐计算方式为：

```text
ops/s = total_g_times / 60
```

## 7. 结果解释

### 7.1 本次结果说明了什么

本次结果给出的是 `no-hook` 场景下的基础吞吐能力，也就是：

- 不开启真实 hook 处理逻辑时
- 该类 `malloc/free` 工作负载在当前 Linux 服务器上的运行上限

这组数据可作为后续 `with-hook` 对照测试的 baseline。

### 7.2 为什么单线程反而更高

从结果看：

- `1` 线程吞吐较高，约 `8.48M ops/s`
- `1 -> 2 -> 4` 线程阶段，总吞吐先下降
- `4 -> 8 -> 16 -> 32 -> 64 -> 80 -> 96` 阶段，总吞吐逐步回升
- `80` 与 `96` 线程结果已经非常接近，说明平台进入饱和区

这通常意味着多线程下存在额外开销，例如：

- 分配器竞争
- 调度开销
- CPU 共享资源竞争
- 缓存一致性与同步相关损耗

因此，本次结果反映的是：

- 单线程时，共享竞争最少，因此吞吐较高
- 进入低线程并发后，共享竞争和调度开销先把总吞吐拉低
- 随着线程数继续增加，更多 CPU 核心参与执行，总吞吐重新上升
- 当线程数达到 `80~96` 时，总吞吐接近平台饱和

因此，多线程结果未线性扩展是可以解释的，并不一定代表测试出错。

### 7.3 为什么 4 线程补跑了两次

初始 3 次结果中，`4` 线程组波动明显大于其他组。为了让结论更稳，本次额外补跑了 `run4` 和 `run5`，最终使用 `5` 次平均值作为 `4` 线程档位结果。

### 7.4 为什么 2 线程结果做了异常值处理

`2` 线程结果中，`run2 = 243,070,909` 明显低于其他 5 次结果，且与整体趋势不一致，因此判断其受到环境干扰的可能性较大。

为了避免异常值拉低结论，最终采用剔除该异常值后的 `5` 次有效平均值：

- avg total_g_times = `475,141,907`
- avg ops/s = `7,919,032`

### 7.5 为什么 80 和 96 线程值得测

测试机器共有 `80` 个逻辑 CPU，因此：

- `80` 线程对应“接近平台并行上限”
- `96` 线程对应“略高于平台并行上限”

结果显示：

- `80` 线程：约 `12.88M ops/s`
- `96` 线程：约 `12.97M ops/s`

两者差异很小，说明在当前 workload 下，平台已经进入吞吐饱和区，继续增加线程数带来的收益已经很有限。

## 8. 本次测试不包含的内容

这次测试不是以下内容：

- 不是完整 OpenHarmony 设备侧 `native_hook` 端到端链路测试
- 不是 `native_daemon + shared memory + eventfd + 真正 hook` 的完整开销测试
- 不是最终的 `with-hook` overhead 测试

本次测试仅可表述为：

> 基于 `native_hook` 仓库自带测试程序，在 Linux 服务器上完成的 `no-hook baseline` 吞吐测试。

## 9. 当前结论

当前已经得到一版可复用的 baseline 数据：

- `1` 线程：约 `8.48M ops/s`
- `2` 线程：约 `7.92M ops/s`（剔除 1 次异常值后）
- `4` 线程：约 `5.42M ops/s`
- `8` 线程：约 `6.05M ops/s`
- `16` 线程：约 `6.37M ops/s`
- `32` 线程：约 `6.63M ops/s`
- `64` 线程：约 `11.04M ops/s`
- `80` 线程：约 `12.88M ops/s`
- `96` 线程：约 `12.97M ops/s`

这组数据可以作为后续评估 hook 开销的参考基线。

## 10. 后续建议

建议下一步继续做以下两类工作之一：

### 方案 A：继续补充 baseline

- 增加不同 `malloc size`，例如 `64B`、`256B`
- 继续观察不同分配粒度对吞吐的影响

### 方案 B：进入对照实验

- 在更完整的环境中启用真实 `hook`
- 采集 `with-hook` 场景下的吞吐
- 与本报告中的 `no-hook baseline` 进行对比
- 计算 overhead

常见对比方式：

```text
overhead = (baseline_ops - hook_ops) / baseline_ops
```
