## Bug 修复详情

### 死锁根因

`hook_writer.cpp::RecordStackWriterSubAblationAllocThreadLocal`（~line 1106）：

```
stack_writer_.Lock()       → pthread_mutex_lock(&inner_mutex_)   // 第一次锁
  stack_writer_.Write()    → pthread_mutex_lock(&inner_mutex_)   // 第二次锁 → 死锁
```

`inner_mutex_` 是 `PTHREAD_MUTEX_DEFAULT`（非递归），同一线程重入 → undefined behavior。
在 Linux glibc 上表现为严重的性能退化（而非完全卡死），这就是为什么旧 sub=34/35 数据
跑出了 8-32s 的极慢数字但没有完全 hang。

### 修复

删除外部 `Lock()/Unlock()` 调用（共 4 行）。`Write()` 内部自行管理锁，测量点
（`HotpathProfileAdd`）仍然包裹了包含内部锁开销的完整 write 操作。

### 副作用

旧 sub=34/35 benchmark 数据（在 double-lock UB 下采集）全部作废，需要重新采集。

## 分析

### 逐层拆解（以 1T 为例）

```
sub=34  ring write + inner_mutex_  = 0.28s
sub=35  + eventfd 通知             = 0.82s  → delta = 0.54s（eventfd syscall）
sub=36  + consumer 消费            = 0.84s  → delta = 0.02s（consumer 几乎免费）
```

### 线程扩展特征

- **sub=34**（纯锁）：16T/1T = 2.6x — `inner_mutex_` 竞争随线程增加
- **sub=35**（锁+通知）：16T/1T ≈ 1.0x — eventfd 无竞争，syscall 开销摊到多线程反而更平
- **sub=36**（全链）：跨线程几乎平坦（0.79-0.88s）— consumer drain 不引入新竞争

### 关键发现：per-record StackWriter 比 batch64 HookWriter 更快

| 方案 | 1T | 16T |
|---|---|---|
| batch64 + HookWriter 全局 mutex (sub=33) | 1.24s | 1.37s |
| per-record + StackWriter 内锁 (sub=36) | **0.84s** | **0.88s** |

StackWriter 临界区只有 ring write 一步操作；HookWriter 全局 mutex 还包裹 tracking、
thread name 等。架构对齐的 StackWriter 内锁比原来的全局锁轻一个数量级。
