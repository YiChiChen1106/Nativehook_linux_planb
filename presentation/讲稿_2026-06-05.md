## 第 2 页：上次组会反馈闭环

上次组会黄总提了四个问题，我逐一回应一下。

第一个，"4T比1T多1.8秒，怀疑有额外热点"——这个已经定位了，多出来的时间是 ring 共享状态竞争导致的，包括 ring index 检查、记录拷贝、atomic publish 这几步。后来我做了批量发布，把 4 线程从 2.72 秒压到了 0.86 秒，不仅消除了差距，甚至反超了单线程。这个热点找到了，也解决了。

第二个，"跑一下 perf 收集"——我没有直接用 perf，而是用 ablation 的方法替代了。我新增了两个 sub-stage，34 和 35，分别测量 ring write 加内部锁的开销、以及 eventfd 通知的开销。这个粒度比 perf 更细，能拆到模块内部的具体操作。

第三个，"fork Gitee 上 native hook 到公司 GitLab"——这个已经做完了。仓库在 gitlab.youtune.tech/cychi/cyc_nativehook，五个核心函数——malloc、calloc、realloc、aligned_alloc、free——全部加上了 producer 端的关键优化。具体优化内容后面展开讲。

第四个，"4线程 eBPF 异常快"——先解释一下为什么 4 线程会比单线程快。因为我们用的是固定总负载的指标，不考虑共享路径竞争的话，多线程并行处理本来就该比单线程花的时间短。上次 4T 数据就体现了这个并行优势。这次我补了 8 线程和 16 线程的实验，三次重复。结果发现 eBPF 在高线程下不仅没退化，反而反超了 LD_PRELOAD，8T 快 2.5 倍，16T 快 3 倍。原因是 eBPF 的 per-CPU ringbuf 没有共享锁竞争，线程越多优势越大。具体数据后面有一页专门讲。

## 第 3 页：Writer/Ring Impact 拆解实验

先说一下这个方法是怎么做的。用一个环境变量 LNHV1_SUBABLATION_STAGE 控制 hook 内部的执行路径。每个 sub-stage 对应热路径上的一个操作步骤，28 是什么都不做，29 是只拿锁，30 在锁内加 ring index 检查，31 再加记录拷贝，32 再加 atomic write_index 发布，33 再加 eventfd 通知和 consumer 交互。每一步都在前一步基础上叠加一项操作，用勾 hook 的耗时减去 baseline 的耗时，就是这步操作的新增开销。

看 8 线程这一列：仅拿锁加 tracking 只多了 0.85 秒，加上 ring index 检查就变成了 1.92 秒，再加上记录拷贝跳到了 2.85 秒，加上 atomic write_index 是 3.10 秒，最后加上 eventfd 通知和 consumer 交互是 3.12 秒。

所以关键结论有两个。第一，多线程额外开销主要来自 ring index 检查和 atomic publish，这些是共享状态的竞争，线程越多越严重。第二，通知和 consumer 交互是第二大开销源，在 8 线程、16 线程下显著抬头。

## 第 4 页：Producer 端两项优化

第一项，记录填充移出写入锁。原来的流程是先拿锁，然后在锁里面填 record 的各个字段——type、addr、size、pid、tid、timestamp，还有每 1000 次更新一次的线程名。这些操作都不碰共享状态——pid 和 tid 是线程本地缓存，timestamp 就是调 clock_gettime——放在锁内只会白白延长其他人等锁的时间。所以我把填充全部移到拿锁之前：先填充，再拿锁，锁内只做写 ring 和通知。4T 从 3.51 秒降到 2.72，省了 22.5%。

这个优化在 OpenHarmony 真实代码里同样适用——真实代码的 hook_malloc 里 FpUnwind 栈回溯和 StackRawData 填充都在 weakClient.lock() 之后做，我已经在 GitLab fork 上把这几步移到了锁前面。

第二项，Stage 6 批量发布。原来每一条 record 都要完整走一遍拿锁、写 ring、发布索引、通知。我加了一个线程本地缓冲区，每条 record 先写进缓冲区，攒满 batch_size 条后一次排空——拿一次锁、批量写、一次发布索引、一次通知。线程退出时析构函数自动排空。batch_size 通过环境变量控制，默认 0 关闭。效果上，1T 从 1.55 降到 1.22，8T 从 3.12 降到 1.28，16T 从 3.41 降到 1.34。

这项在真实代码里的意义更大。真实代码每条 SendStackWithPayload 都会触发 ShareMemoryBlock 内部的 PutWithPayloadTimeout，每次调用都要拿一次内部锁。批量发布把逐记录的锁操作压成了逐批次，和原型上一样的效果。这个也已经移植到 GitLab fork 上了。

## 第 5 页：批量发布 — 工作原理

左边是优化前：每条 record 进来都独立走一遍蓝色矩形框——先拿锁，再写 ring，再 eventfd 通知。三条记录就三次锁、三次通知。

右边是优化后：64 条 record 全部先写进虚线框的缓冲区，缓冲区里只做计数。等攒满 64 条才触发一次排空——橙色矩形框：一次拿锁、一次写入 64 条、一次发布索引、一次通知。

经过优化：100 万次拿锁压缩到 1.5 万次，100 万次 atomic publish 压缩到 1.5 万次，~5 万次 eventfd 压缩到 ~780 次。具体性能数据接下来一页。

## 第 7 页：eBPF 高线程验证

上次组会发现 4 线程下 eBPF 只比 optimized LD_PRELOAD 慢 0.18 秒，当时觉得差距不大但没有反超。这次补了 8 线程和 16 线程的实验，各跑了三次重复。

结果非常一致——8 线程下 eBPF 平均 1.71 秒，LD_PRELOAD optimized 是 4.27 秒，eBPF 快 2.5 倍。16 线程下更明显，eBPF 平均 1.47 秒，LD_PRELOAD 是 4.39 秒，快 3 倍。

原因是 eBPF 用的是 per-CPU ringbuf，每个 CPU 有自己的 ring buffer，根本没有跨线程的共享锁竞争。LD_PRELOAD 这边即使用了批量发布，仍然有逐批次的 writer 锁，线程多了竞争还是会上来。

所以结论要修正——低线程场景下 LD_PRELOAD 更好，高线程场景下 eBPF 的天花板更高。

## 第 8 页：栈写入器 模块级瓶颈定位

这是新加的 ablation sub-stage 34 和 35 的数据。之前 28 到 33 测的是整个 hook_writer 层的开销，ring write 和 notify 混在一起。我这次在原型里加了 stack_writer 模块，内部带了锁来模拟真实代码的 ShareMemoryBlock，然后拆成了两步测量。

34 是 write_only，只测 ring write 加内部锁，1T 是 8.60 秒。35 是 flush_only，加了 eventfd 通知，1T 是 9.15 秒。两者差距只有 0.55 秒——也就是说，ring write 内锁占了总开销的 97%，eventfd 只占 3%。内锁是明确的主瓶颈。

而且逐记录锁竞争随线程数恶化：8T 的 write 比 1T 慢了 3.2 倍。batch64 的效果非常显著——1T 从 9.15 秒压到 1.24 秒，8T 从 23.94 秒压到 1.26 秒，19 倍的提升。

这些数据可以直接对标到真实 OpenHarmony 代码的 ShareMemoryBlock::PutWithPayloadTimeout 和 EventNotifier::Post 两个函数。

## 第 9 页：原型 ↔ OpenHarmony 架构对齐

左边绿色栏是已经对齐的模块。malloc 入口和重入保护是原型原来就有的。地址追踪、栈写入器写共享内存、栈写入器通知消费者——这三项是这次新加的，对应真实代码的 AddAllocAddr、WriteWithPayloadTimeout、Flush→EventNotifier::Post。加它们就是为了让原型和真实代码的热路径结构一一对应。

右边粉色栏列出了仍然存在的差异。FpUnwind 栈回溯是 aarch64 专用指令，原型跑在 x86 上没法实现。StackRawData 的格式也不一样，原型是简化版，真实代码里有 ip 数组和 jsChainId。record 的大小在真实代码里根据 fpunwind 模式动态计算，原型是固定大小。构建系统不一样，原型用 CMake 可以本地编译测试，真实代码用 GN 加完整的 OH SDK，目前还没搭起来。共享内存层多了一层 ShareMemoryBlock 封装。consumer 交互多了一层 EventNotifier 加心跳检测。

这些差异不影响原型上做优化假设——因为热路径的核心结构已经对齐了。但移植到真实代码后需要编译验证才能确认优化在完整环境下同样有效。

## 第 10 页：OpenHarmony Fork — 移植状态

仓库在公司的 GitLab 上，master 分支。五个核心函数全部加了两项优化：record-fill-before-lock 和 batch publish。

hook_malloc 是主分配路径，hook_calloc 对应 C++ new 的底层调用，hook_realloc 对应 vector 扩容——它比较特殊，一次 realloc 会发两条记录，一条 free 旧地址、一条 alloc 新地址，批量发布里也处理了。aligned_alloc 只加了 fill-before-lock，没做批量发布。原因是它调用频率远低于 malloc、calloc、realloc，优化收益太小，不值得加那一套 buffer 逻辑。hook_free 是释放路径，这周刚加上的。

没移植的两项：PID/TID cache 在真实代码里已经用 pthread_getspecific 实现了，不需要再加；tracking fallback 在真实代码里不存在对应的路径，不适合移植。

目前的状态是代码已经推到分支上，等 OpenHarmony 编译环境到位就可以跑 benchmark 验证。

## 第 11 页：经验教训 & 三步预检方法

左边是之前做的两个优化，后来发现是无效的。PID/TID cache 这个，真实代码已经用 pthread_getspecific 缓存了 TID，PID 也是 atomic load 一次，不需要重复做。优化方向是对的，但人家早有了。tracking fallback 的问题更大——它是为了解决原型里 per-thread 查表的多线程竞争，但真实代码的 free 记录是直接发给 daemon 服务端做匹配的，producer 端根本不维护那个表，也不走那条路径。优化了一个只在简化原型里存在的瓶颈，方向就搞错了。

右边是我总结的三步预检法，已经写进了 AGENTS.md。第一步，画映射表——原型每个模块对应真实代码的哪个函数、第几行。第二步，结构对齐——原型缺了哪些模块就补上，保证实验数据能精确对标。第三步，三问预检——任何优化方案在动手之前，先回答三个问题：改原型的哪个模块？对应真实代码的哪个函数？真实代码走不走这条路径？有一个答不上来，就不动手。
