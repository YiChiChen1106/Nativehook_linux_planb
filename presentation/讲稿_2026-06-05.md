## 第 3 页：Writer/Ring Impact 拆解实验

先说一下这个方法是怎么做的。原理很简单——用一个环境变量 LNHV1_SUBABLATION_STAGE 控制 hook 内部的执行路径。每个 sub-stage 对应热路径上的一个操作步骤，28 是什么都不做，29 是只拿锁，30 在锁内加 ring index 检查，31 再加记录拷贝，32 再加 atomic write_index 发布，33 再加 eventfd 通知和 consumer 交互。每一步都在前一步基础上叠加一项操作，用勾 hook 的耗时减去 baseline 的耗时，就是这步操作的新增开销。所以这个表是从下往上叠加的。

看 8 线程这一列：仅拿锁加 tracking 只多了 0.85 秒，加上 ring index 检查就变成了 1.92 秒，再加上记录拷贝跳到了 2.85 秒，加上 atomic write_index 发布是 3.10 秒，最后加上 eventfd 通知和 consumer 交互是 3.12 秒。

关键结论有两个。第一，多线程额外开销主要来自 ring index 检查和 atomic publish，这些是共享状态的竞争，线程越多越严重。第二，notify 和 consumer 交互是第二大开销源，在 8 线程、16 线程下显著抬头。

## 第 4 页：Producer 端两项优化

第一项，Record fill outside writer mutex。原来的流程是先拿锁，然后在锁里面填 record 的各个字段——type、addr、size、pid、tid、timestamp，还有每 1000 次更新一次的 thread name。这些操作都不碰共享状态——pid 和 tid 是 thread-local cache，timestamp 就是调 clock_gettime——放在锁内只会白白延长其他人等锁的时间。

所以我把填充全部移到拿锁之前：先 fill，再 lock，锁内只做写 ring 和通知。4T 从 3.51 秒降到 2.72，省了 22.5%。核心逻辑是锁本身不是瓶颈，锁内做的事情决定了竞争窗口的大小。

第二项，Batch Publish。这是核心优化。原来每一条 record 都要完整走一遍拿锁、写 ring、publish、通知。我加了一个 per-thread buffer，每条 record 先写进 buffer，攒满 batch_size 条后一次 flush——拿一次锁、批量写、一次 publish、一次通知。线程退出时析构函数自动排空。batch_size 通过环境变量控制，默认 0 关闭。效果上，1T 从 1.55 降到 1.22，8T 从 3.12 降到 1.28，16T 从 3.41 降到 1.34。

## 第 5 页：Batch Publish — 工作原理

这页用一张图对比两种模式。

左边是 Per-Record 优化前：每条 record 进来都独立走一遍蓝色矩形框——先拿锁，再写 ring，再 eventfd 通知。三条记录就三次锁、三次通知，推到底部说明"每 percord 走一遍拿锁、写 ring、通知"。

右边是 Per-Batch 优化后：64 条 record 全部先写进左边虚线框的 buffer，buffer 里只做 count++。等攒满 64 条才触发一次 flush——橙色矩形框：一次拿锁、一次写入 64 条、一次 atomic publish、一次 notify。

底部橙色横幅是压缩比：100 万次拿锁 → 1.5 万次，100 万次 atomic publish → 1.5 万次，~5 万次 eventfd → ~780 次。直接用数字比文字直观。具体性能数据接下来一页。

## 第 7 页：eBPF 高线程验证

这页是上次遗留问题的最终结论。上次组会发现 4 线程下 eBPF 只比 optimized LD_PRELOAD 慢 0.18 秒，当时觉得差距不大但没有反超。这次补了 8 线程和 16 线程的实验，各跑了三次重复。

结果非常一致——8 线程下 eBPF 平均 1.71 秒，LD_PRELOAD optimized 是 4.27 秒，eBPF 快 2.5 倍。16 线程下更明显，eBPF 平均 1.47 秒，LD_PRELOAD 是 4.39 秒，快 3 倍。

原因是 eBPF 用的是 per-CPU ringbuf，每个 CPU 有自己的 ring buffer，根本没有跨线程的共享锁竞争。LD_PRELOAD 这边即使用了 batch publish，仍然有 per-batch 的 writer 锁，线程多了竞争还是会上来。

所以结论要修正——低线程场景下 LD_PRELOAD 更好，高线程场景下 eBPF 的天花板更高。

## 第 8 页：StackWriter 模块级瓶颈定位

这页是新加的 ablation sub-stage 34 和 35 的数据。之前 28 到 33 测的是整个 hook_writer 层的开销，ring write 和 notify 混在一起。我这次在原型里加了 stack_writer 模块，内部带了锁来模拟真实代码的 ShareMemoryBlock，然后拆成了两步测量。

34 是 write_only，只测 ring write 加内部锁，1T 是 8.60 秒。35 是 flush_only，加了 eventfd 通知，1T 是 9.15 秒。两者差距只有 0.55 秒——也就是说，ring write 内锁占了总开销的 97%，eventfd 只占 3%。内锁是明确的主瓶颈。

而且 per-record 锁竞争随线程数恶化：8T 的 write 比 1T 慢了 3.2 倍。batch64 的效果非常显著——1T 从 9.15 秒压到 1.24 秒，8T 从 23.94 秒压到 1.26 秒，19 倍的提升。

这些数据可以直接对标到真实 OpenHarmony 代码的 ShareMemoryBlock::PutWithPayloadTimeout 和 EventNotifier::Post 两个函数。

## 第 9 页：Prototype ↔ OpenHarmony 架构对齐

这页展示的是原型和真实代码在热路径上的模块映射。左边是原型，右边是 OpenHarmony 的 hook_client.cpp。

上半部分——malloc 入口、re-entry guard、record 填充——原来就有。下半部分标了 [NEW] 的是我这周新加的。address_handler 对应真实代码的 AddAllocAddr，stack_writer 的 write 对应 WriteWithPayloadTimeout，flush 对应 Flush 再到 EventNotifier::Post。

## 第 10 页：OpenHarmony Fork — 移植状态

这是黄总上次提的 fork 任务的完成情况。仓库在公司的 GitLab 上，master 分支。五个核心函数全部加了两项优化：record-fill-before-lock 和 batch publish。

hook_malloc 是主分配路径，hook_calloc 对应 C++ new 的底层调用，hook_realloc 对应 vector 扩容——它比较特殊，一次 realloc 会发两条记录，一条 free 旧地址、一条 alloc 新地址，batch 里也处理了。aligned_alloc 只加了 fill-before-lock，没做 batch。hook_free 是释放路径，这周刚加上的。

没移植的两项：PID/TID cache 在真实代码里已经用 pthread_getspecific 实现了，不需要再加；tracking fallback 在真实代码里不存在对应的路径，不适合移植。这也恰好是我之前走弯路的两个优化。

目前的状态是代码已经推到分支上，等 OpenHarmony 编译环境到位就可以跑 benchmark 验证。

## 第 11 页：经验教训 & 三步预检方法

聊一下这周的两个教训。

左边是之前做的两个优化，后来发现是无效的。PID/TID cache 这个，真实代码已经用 pthread_getspecific 缓存了 TID，PID 也是 atomic load 一次，不需要重复做。优化方向是对的，但人家早有了。tracking fallback 的问题更大——它是为了解决原型里 per-thread 查表的多线程竞争，但真实代码的 free 记录是直接发给 daemon 服务端做匹配的，producer 端根本不维护那个表，也不走那条路径。优化了一个只在简化原型里存在的瓶颈，方向就搞错了。

右边是我总结的三步预检法，已经写进了 AGENTS.md。第一步，画映射表——原型每个模块对应真实代码的哪个函数、第几行。第二步，结构对齐——原型缺了哪些模块就补上，保证实验数据能精确对标。第三步，三问预检——任何优化方案在动手之前，先回答三个问题：改原型的哪个模块？对应真实代码的哪个函数？真实代码走不走这条路径？有一个答不上来，就不动手。

这个方法能保证以后不再出现原型里做得热火朝天、移植时发现不适用的情况。
